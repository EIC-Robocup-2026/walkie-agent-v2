# GPSR Challenge — Design

> Rulebook 5.3 (RoboCup@Home 2026). Spec: [`docs/GPSR.pdf`](GPSR.pdf).
> Scaffold: [`tasks/GPSR/`](../tasks/GPSR). Built on the task framework in
> [`tasks/base.py`](../tasks/base.py); the reference task is [`tasks/HRI/`](../tasks/HRI)
> and the closest sibling design is [`docs/RESTAURANT_DESIGN.md`](RESTAURANT_DESIGN.md).
>
> Command grammar derived from the official
> [RoboCupAtHome/CommandGenerator](https://github.com/RoboCupAtHome/CommandGenerator)
> (the rulebook mandates it). Arena nouns (rooms/objects/names) come from the
> per-competition CompetitionTemplate, loaded as config like Restaurant's poses.
>
> Status: **design** — agreed direction, not yet fully implemented. The current
> `tasks/GPSR/` is a thin "throw the raw command at the agent" placeholder that
> this design replaces.

---

## 1. The task, and where the points are

The operator issues **three arbitrary commands**, generated by the official
CommandGenerator then **re-phrased by an LLM up to 3× (getting simpler each
time)**. The robot starts outside the arena; when the door opens it drives to the
**Instruction Point**, takes the command(s) — it chooses whether to receive them
all at once (enables the interleave bonus) or one-by-one (return to the operator
between each) — executes them, and returns to the Instruction Point. The arena is
in nominal configuration; commands may send the robot out of the room.

Score map (total **1490**) — this is the strategic spine of the whole design:

| Action | Points | Needs |
|---|--:|---|
| Understand the spoken command | 3×80 = 240 | **STT (no arm, no nav)** |
| Demonstrate a plan has been generated | 3×100 = 300 | **parse + speak a plan (no arm, no nav)** |
| Solve the first / second / third command | 3×250 = 750 | nav + perception + (often) manipulation |
| Interleaved Task Bonus | 200 | scheduler, all-3-at-once, *meaningful* |
| *(penalties)* | — | custom operator 3×−20, **rephrasing 6×−30**, bypass STT 3×−50 |

Partial scoring applies: "solving part of a command" earns a percentage of the 250.

**The load-bearing insight: 540 points (36%) are draw-independent and
hardware-independent.** Understand all three commands (240) and *speak a correct
plan* for each (300) — pure STT + parse + TTS, no arm, no navigation, no object
contact, and **whatever** the three drawn commands turn out to be. This is the
safest, highest-leverage budget in the test — but it is **parser-dependent**, not
free: a wrong plan likely forfeits both the 300 *and* tanks that command's 250
(§5.1). So parser correctness, not hardware, is what "locks" this 540 — which is
why offline parser coverage is the Phase-0 gate (§10).

The *solve* budget (750) is a different story: **the operator draws the three
commands, not us.** Navigation-class commands (`go to`, `find`, `count`, `tell`,
`greet`) solve with no arm — but if the draw is heavy on `bringMeObjFromPlcmt` /
`takeObjFromPlcmt`, **manipulation can gate ~500 of the 750**. So "no-arm" is a
*gamble on the draw*, not a safe floor; the draw-independent floor is the 540.
Manipulation is still phased last (hardest, least reliable, blocked on arm
calibration), but it must be pushed harder than "a minority of the score" —
expect it to decide a typical run's solve total.

**Time budget is brutal and a first-class design driver.** **7:00 total for
three commands** (~2:20 each, incl. travel) vs. Restaurant's 15:00 for two. The
referee pauses the clock only once the robot is back at the Instruction Point.
Consequences baked into the design: prefer fast, reliable command categories over
ambitious slow ones; never let understanding stall (each rephrasing is −30, six
possible = −180); keep the plan→execute path short.

## 2. Why GPSR is different from HRI / Restaurant

| | HRI / Restaurant | GPSR |
|---|---|---|
| Flow | a *fixed* (HRI) or *reactive but bounded* (Restaurant) physical procedure | **arbitrary** — any of ~27 command templates over the arena, in any order |
| What's scored | doing the physical steps | **understanding + an explicit plan + doing** — the *plan itself* is 300 pts |
| Planning | hardcoded `SubTask` sequence | must be **generated per command at run time** |
| Breadth | one capability cluster (seating / waiting) | **all of them** — nav, find, manipulate, follow, guide, count, tell, greet |

Two facts flow from this:

1. **GPSR cannot be a single hardcoded `SubTask` chain** the way HRI is. The
   *outer envelope* is fixed (go to instruction point → receive → execute →
   return) but the *inner* execution of each command must be assembled at run
   time from the operator's words.
2. **"Demonstrate a plan has been generated" (300 pts) means the plan must be a
   real, articulable artifact** — and given Walkie's "no plain text output"
   contract, *demonstrable* = **`speak()` the plan aloud**. A design that just
   acts without surfacing a plan forfeits 20% of the score. This single
   requirement is what pushes us away from the current "throw the string at the
   agent and let it act" scaffold.

## 3. The command grammar → primitive vocabulary + world model

The generator's ~27 templates (primary + follow-up) collapse to a **small set of
atomic primitives**. This separation is the core of the design (per the
adversarial review): **the grammar/verbs are stable and derived now; the nouns
(rooms, objects, names, gestures) are arena-specific and config-loaded.**

### 3.1 Stable grammar → atomic primitives

Every generated command is one or two clauses over these verbs (generator
category names in parentheses):

| Atomic primitive | Args | Generator categories it covers |
|---|---|---|
| `navigate` | location \| room \| placement | goToLoc; the implicit "go to" prefix of nearly every other command |
| `find_object` | object, location? | findObj, findObjInRoom |
| `find_person` | descriptor (name \| gesture \| pose \| clothing), room? | findPrs, findPrsInRoom, meetName, meetPrsAtBeac |
| `pick` | object, placement? | takeObjFromPlcmt |
| `place` | object, placement | placeObjOnPlcmt |
| `deliver` | object, recipient (me \| person-descriptor \| name@loc) | bringMeObjFromPlcmt, deliverObjToMe, deliverObjToPrsInRoom, deliverObjToNameAtBeac |
| `follow` | person, to-room? | followPrsAtLoc, followNameFromBeacToRoom, followPrsToRoom |
| `guide` | person, from-loc?, to-loc | guideNameFromBeacToBeac, guidePrsFromBeacToBeac, guideClothPrsFromBeacToBeac, guidePrsToBeacon |
| `count` | what (objects-on-placement \| persons-in-room), where | countObjOnPlcmt, countPrsInRoom |
| `get_person_info` | which (name \| pose \| gesture \| clothing) | feeds tellPrsInfoInLoc ("object/people **feature recognition**") |
| `get_object_property` | which (size \| weight \| category \| color) | feeds tellObjPropOnPlcmt ("object feature recognition") |
| `say` | info (a fact, an answer, a remembered attribute) | tellPrsInfoInLoc, tellObjPropOnPlcmt, talkInfoToGestPrsInRoom |
| `greet` | person (name \| clothing), room | greetNameInRm, greetClothDscInRm |

A command like *"go to the kitchen, find a coke, and bring it to me"* parses to
`[navigate(kitchen), find_object(coke, kitchen), pick(coke), deliver(coke, me)]`.

Person **descriptors** are the union {name, gesture, pose, clothing+color} — and
this is exactly HRI's wheelhouse (§5.4). Object **properties** (size/weight/
category/color) and person **features** (pose/gesture/clothing) are the
"feature recognition" the rulebook focus calls out.

### 3.2 Arena nouns → config-driven world model

These are announced per competition (CompetitionTemplate: `rooms`, `locations`,
`objects`, `names`, `gestures`). They are **data, not code** — loaded like
Restaurant's poses (§8):

- **rooms** — name → nav pose (a representative point to stand in the room)
- **locations / placements / beacons** — name → nav pose + which room + (for
  manipulation) a surface height
- **objects** — name → category, and known size/weight/color where the template
  gives them (for `get_object_property` without re-measuring)
- **names** — the person names meet/greet/guide can reference
- **gestures / poses** — the recognizable set (waving, raising L/R arm, pointing
  L/R, sitting, standing, lying) mapped to keypoint heuristics (reuse Restaurant
  `is_calling`'s style)

The parser maps an utterance → a typed plan over §3.1; the world model **grounds
the nouns** (kitchen → pose, coke → category) so the executor can act.

## 4. Architecture — two-tier hybrid (recommended)

Following HRI/Restaurant philosophy (deterministic control, **LLM only at the
edges**), GPSR is a **two-tier executor** wrapped in the fixed rulebook envelope.

```
┌────────────────────────────────────────────────────────────────────┐
│ OUTER ENVELOPE (fixed SubTask chain — like the current scaffold)     │
│   GoToInstructionPoint → ReceiveCommands → ExecuteCommands → Return  │
├────────────────────────────────────────────────────────────────────┤
│ PARSE  (LLM, at the edge)                                            │
│   utterance → typed Plan = [PlanStep(primitive, **grounded args)]    │
│   over the §3.1 vocabulary + §3.2 world model                        │
├────────────────────────────────────────────────────────────────────┤
│ DEMONSTRATE  →  speak() the plan aloud   (scores the 300)            │
├────────────────────────────────────────────────────────────────────┤
│ EXECUTE  (per step)                                                  │
│   Tier 1 (primary): deterministic dispatch — primitive → a skill in  │
│      tasks/GPSR/skills.py, reusing HRI + Restaurant skills.          │
│   Tier 2 (fallback): a step the parser couldn't ground, or a skill   │
│      that failed, is handed to the agent stack (delegate_to_*) as a   │
│      natural-language sub-task — the escape hatch for the long tail.  │
├────────────────────────────────────────────────────────────────────┤
│ LLM also used ONLY for: parse, word the plan, answer `say`/`tell`    │
│   questions, decide interleave schedule (last).                      │
└────────────────────────────────────────────────────────────────────┘
```

### Why this, and not the alternatives

- **Not pure-agent (the current scaffold).** Handing the raw string to
  `walkie_agent.invoke` is the *one* approach the rulebook actively punishes: it
  surfaces **no demonstrable plan** (−300 vs. our design), puts the LLM inside
  the multi-step physical control loop (slow + unreliable against a 7-min clock),
  and inherits the agent's coarse primitives — `delegate_to_actuator("pick up the
  cup")` is a **silent no-op today** because `command_arm` calls SDK methods that
  don't exist (verified in `agents/actuator_agent/tools.py`). It also matches
  neither the user's "follow HRI" instruction nor the repo's established split.
- **Not pure hardcoded state machine (HRI-style).** Impossible — the command is
  unknown until run time.
- **The hybrid keeps the LLM where it's strong (understanding language, deciding,
  answering) and out of where it's weak (real-time, multi-step physical
  control)** — verbatim the Restaurant §4 decision — while making the plan a
  first-class artifact and reusing already-validated skills. The Tier-2 agent
  fallback means an exotic phrasing degrades to *flexible* rather than *failed*.

This is a recommendation, not a settled fact — the main fork is "how much do we
lean on Tier 2." See §11.

## 5. Load-bearing principles

### 5.1 The plan must be spoken — and it's only as good as the parse
"Demonstrate a plan has been generated" is judged by the robot **articulating**
it. Because the typed `Plan` is a real object, wording it is a trivial
deterministic render ("First I will go to the kitchen, then find a coke, then
bring it to you") with an optional LLM polish. **Never skip the speak step** —
it's 100 pts/command for ~0 *hardware* risk, and it doubles as the operator's
confirmation that understanding was correct. But the 300 is **not** unconditional:
a plan spoken from a wrong parse can forfeit the points and mislead execution, so
the real risk here is the parser, not the speaking. That makes offline parser
accuracy the thing to measure before anything else (§10 Phase 0). *(Open check:
confirm a wrong spoken plan still scores "understand the command" — if not,
parser accuracy compounds across both the 240 and the 300.)*

### 5.2 Minimize rephrasings — understanding is worth more than precision
Each requested rephrasing is **−30** (six possible = −180), and a custom operator
is −20/command + caps you to the three rephrased forms. So: parse **optimistically**
from the first utterance, ground what you can, and only re-ask when the parse is
empty — not to "confirm." A partially-grounded plan that scores partial credit
beats a rephrasing penalty. (Mirrors HRI's "no non-essential questions" instinct.)

### 5.3 Design against drift — re-detect, don't trust stored coordinates
Same as Restaurant §5.1. The world-model pose gets the robot *near* a room/
placement; the **final docking and all manipulation act on a fresh on-arrival
detection**, never a coordinate captured minutes earlier. Reuse
`CameraSnapshot.bbox_world_xy` (HRI/Restaurant lift).

### 5.4 People commands reuse HRI's machinery — `people` is NOT None
Unlike Restaurant (`people=None`), GPSR is thick with person commands (meet,
greet, find-by-gesture/clothing, guide, follow, "tell the person…"). So the GPSR
`TaskContext` gets a `PeopleStore`, and we reuse `tasks/HRI/identity.py`
(`enroll_*`, `locate_people`) and Restaurant's `is_calling`/gesture keypoint
heuristics. **This is plausibly why the task is framed as "follow HRI."**

### 5.5 The interleave bonus is the *last* thing — but bias toward all-at-once
200 of 1490, awarded only when all three commands are taken at once **and** the
interleaving is *meaningful* (e.g. pick up object for command 3 while passing
through on command 1). MVP is **strictly serial, one command at a time**; the LLM
scheduler that earns the bonus comes only after everything else works. Note the
rulebook says the robot *decides and advises* how commands are issued — so the
greeting should **actively request all three at once** (and fall back to
one-by-one only if STT struggles), keeping the bonus reachable, rather than
passively offering the operator a choice. Even when executed serially, taking all
three up front also saves the round-trips of returning to the operator between
commands — pure time win against the 7-min clock.

## 6. State model

```python
class Primitive(Enum):
    NAVIGATE; FIND_OBJECT; FIND_PERSON; PICK; PLACE; DELIVER
    FOLLOW; GUIDE; COUNT; GET_PERSON_INFO; GET_OBJECT_PROPERTY; SAY; GREET

@dataclass
class PlanStep:
    primitive: Primitive
    args: dict          # grounded against the world model (room→pose, obj→category)
    raw: str            # the clause this came from, for the spoken plan + debugging

@dataclass
class Command:
    id: int                       # 1..3
    utterance: str                # what STT heard
    plan: list[PlanStep]          # from the parser (may be empty → re-ask once)
    status: CmdStatus             # RECEIVED → PLANNED → IN_PROGRESS → DONE/PARTIAL/FAILED
    result_note: str | None       # e.g. the answer spoken, or why it failed
```
Stored on `ctx.data["commands"]: list[Command]` plus `ctx.data["world"]` (the
§3.2 model). The executor walks each command's `plan`; the (later) scheduler
chooses which command's next step to advance for the interleave bonus.

## 7. Primitives → skills (`tasks/GPSR/skills.py`)

Plain functions over `TaskContext`, HRI style. **Reuse** column is the point —
most of GPSR is gluing existing skills behind the parser.

| Primitive | Skill | Reuse / build |
|---|---|---|
| `navigate` | `go_to_named(ctx, name)` | world-model pose + `ctx.goto` (have) |
| `find_object` | `find_object(ctx, obj, where?)` | `walkie_graphs` memory + live `object_detection` re-detect on arrival (have via agent/db) |
| `find_person` | `find_person(ctx, descriptor, room?)` | HRI `locate_people` (name/clothing) + Restaurant `is_calling`-style gesture/pose keypoints 🟡 |
| `pick` | `pick_item(ctx, obj)` | **Restaurant `pick_item`** — reuse; still uncalibrated, `command_arm` is a no-op ⚠️ |
| `place` | `place_item(ctx, placement)` | **Restaurant `serve_item`** generalized ⚠️ |
| `deliver` | `deliver(ctx, obj, recipient)` | navigate + find_person + (place \| handover) 🟡 |
| `follow` | `follow_person(ctx, person, to?)` | **greenfield** — needs a follow primitive (Restaurant §3 flagged this missing) 🔴 |
| `guide` | `guide_person(ctx, person, to)` | navigate while keeping person tracked + arrival announce 🔴 |
| `count` | `count(ctx, what, where)` | navigate + detect + len() 🟢 |
| `get_person_info` | `person_info(ctx, which)` | HRI appearance caption + pose/gesture keypoints + face/name re-ID 🟡 |
| `get_object_property` | `object_property(ctx, obj, which)` | world-model lookup first; else caption/measure 🟡 |
| `say` / answer | `answer(ctx, question)` | `ctx.extract` / LLM Q&A + `ctx.say` 🟢 |
| `greet` | `greet(ctx, person)` | find_person + `ctx.say` 🟢 |

🟢 cheap / 🟡 moderate (mostly reuse) / 🔴 greenfield / ⚠️ blocked on arm calibration.

**Manipulation lives where Restaurant's does.** Resolve the open
`tasks/base.py` shared pick/place question (Restaurant §11): GPSR, Restaurant,
Pick&Place and Laundry all want the same grasp primitive — promote it to
`tasks/base.py` rather than a fourth copy.

## 8. Config (`tasks/GPSR/config.toml`, additive to the scaffold)

The existing scaffold already has `GPSR_INSTRUCTION_POINT_POSE`,
`GPSR_MAX_COMMANDS`, `GPSR_ISSUE_MODE`, `GPSR_START_PERCEPTION`. Add the world
model + knobs:

```toml
# World model (the CompetitionTemplate nouns — announced per competition)
GPSR_ROOMS         # "kitchen=x,y,h; living_room=x,y,h; ..."
GPSR_LOCATIONS     # "dinner_table=x,y,h,room,surface_z; counter=...; ..."
GPSR_OBJECTS       # path/inline: name→category,size,weight,color
GPSR_NAMES         # known person names for meet/greet/guide
GPSR_GESTURES      # recognizable gesture/pose set → keypoint rule ids
# Behaviour
GPSR_STANDOFF_M           # stop distance in front of a person/placement
GPSR_SPEAK_PLAN           # 1 = speak the plan before executing (scores 300) — default on
GPSR_REASK_ON_EMPTY_ONLY  # 1 = only re-ask when parse is empty (minimize −30s)
GPSR_ENABLE_MANIPULATION  # 0 until arm is calibrated (mirrors RESTAURANT_ARM_CALIBRATED)
GPSR_INTERLEAVE           # 0 = serial MVP, 1 = LLM scheduler (bonus, last)
```

(The world-model format can be a richer file like the object table rather than a
flat string — decide alongside the CompetitionTemplate import.)

## 9. Mapping onto the scaffold

- `subtasks.py` — keep the four-step envelope (`GoToInstructionPoint`,
  `ReceiveCommands`, `ExecuteCommands`, `ReturnToInstructionPoint`). **Rework
  `ReceiveCommands`** to parse each command into a typed `Plan` and **speak it**
  (the 300). **Rework `ExecuteCommands`** from "invoke the agent on the raw
  string" to "walk each `Command.plan`, dispatch each `PlanStep` to a skill,
  agent-fallback on miss." Add the §6 dataclasses.
- `parse.py` (new) — the LLM utterance→typed-`Plan` parser + world-model
  grounding. The one genuinely new module vs. HRI's shape.
- `skills.py` (new) — the §7 primitives, importing HRI/Restaurant skills.
- `world.py` (new, or in `parse.py`) — load the §3.2 world model from config.
- `prompts.py` — add the parser schema/instructions and the plan-wording prompt;
  keep the existing spoken lines.
- `run.py` — set `ctx.people = PeopleStore.from_env()` (§5.4); still build the
  `WalkieBrain` for the Tier-2 fallback and keep the background perception loop.
- `config.toml` — add §8.

## 10. Roadmap (scoring-driven, validated on-robot)

- **Phase 0 — the parser + the 540 (all software, no robot):** the highest-value,
  fully-offline-testable phase.
  1. **Import the CompetitionTemplate** (rooms/locations/objects/names) — this one
     import yields *both* the §3.2 world model (nouns to ground) *and* the
     generator's `DATA_DIR`.
  2. **Build `parse.py`** (utterance → typed plan, grounded against the world model)
     and the deterministic plan→speech render.
  3. **Use the official generator as the test oracle.** Run
     `generator.py -d <template>` to emit a large command corpus (templates *and*
     LLM-rephrased simpler variants), feed it through `parse.py`, and measure
     **coverage = fraction that ground to a complete typed plan with no Tier-2
     fallback.** That number is the Phase-0 acceptance gate — there is no on-robot
     test for parse quality, so this offline corpus number *is* the confidence in
     the 540. Wire it as a pytest under `tests/`.
  4. End-to-end dry run with `DISABLE_LISTENING=1`: type a command, read back the
     spoken plan. This phase makes available understand 240 + plan 300 — contingent
     on the coverage number, not on any hardware.
- **Phase 1 — navigation-class solves (no manipulation, ~most of 750):**
  `navigate`, `find_object`, `find_person`, `count`, `say`/answer, `greet`, and
  the deterministic dispatch + Tier-2 agent fallback. Serial. Reuses HRI people
  skills + the scene graph. This is the bulk of the "solve" budget with zero arm
  risk.
- **Phase 2 — manipulation:** `pick`/`place`/`deliver` on Restaurant's
  `pick_item`/`serve_item` (promote to `tasks/base.py`); gated by
  `GPSR_ENABLE_MANIPULATION` until the arm is calibrated (reuse Restaurant's
  calibration checklist). `follow`/`guide` greenfield primitives.
- **Phase 3 — interleave bonus (200):** the LLM scheduler that takes all three at
  once and meaningfully interleaves. Last, exactly like Restaurant §5.3.

## 11. Open questions / risks

- **Tier-1 vs. Tier-2 balance (the main architecture fork).** How much do we
  invest in deterministic skills vs. leaning on the agent fallback? Recommendation:
  deterministic for the common, fast categories (nav/find/count/say/greet — they
  carry Phase 1); agent-fallback for the rare/awkward tail. Revisit once the
  parser's real coverage is measured.
- **Parser robustness to 3× LLM rephrasing.** *Resolved into Phase 0:* the
  operator's words are themselves LLM-rephrased and get *simpler* each time, so the
  parser must not over-fit to canonical generator phrasing — the corpus oracle
  (§10) measures exactly this. Remaining choice: the coverage threshold to call
  Phase 0 done.
- **World-model / CompetitionTemplate format.** *Pulled into Phase 0* (the import
  gives both the world model and the test corpus). Remaining choice: the on-disk
  format (flat config strings vs. a structured file / direct template import).
- **Does a wrong spoken plan still score "understand"?** A judging nuance the PDF
  doesn't settle (§5.1). If not, parser accuracy compounds across the 240 *and*
  the 300 — worth confirming with referees, as it raises the Phase-0 bar.
- **Manipulation home — `tasks/base.py` shared pick/place** (Restaurant §11):
  resolve now so GPSR/Restaurant/Pick&Place/Laundry share one grasp primitive.
- **Arm still un-calibrated and `command_arm` is a no-op** — manipulation
  commands score 0 until Restaurant's Phase-2 calibration lands. Phase the build
  so this blocks the *minority* of points (§1), not the run.
- **7-minute clock** (§1) — measure real per-command wall-clock early; it may
  force dropping slow categories or capping retries harder.
- **Out-of-room navigation** — commands may leave the arena; the world model and
  SLAM map must cover those locations.
- **`say`/`tell` knowledge** — some "tell" commands need general knowledge (the
  day, a joke, the team name). Decide the canned-vs-LLM answer source.

## 12. Implementation status (branch `feat/GPSR`, off `origin/main`)

- **Phase 0 — DONE (offline-verified on the dev box):** the parser + the
  draw-independent 540.
  - `world.py` + `world.toml` — the arena world model (CompetitionTemplate
    vocabulary as default; poses are placeholders to fill ~2h before the test).
    Alias/article/plural-tolerant grounding. Pure, no hardware.
  - `plan.py` — `Primitive`/`PlanStep`/`Plan` typed model + deterministic
    `render_plan_speech` (the spoken plan that scores the 300).
  - `parse.py` — LLM `RawPlan` extraction + pure `ground_plan`/`ground_step`
    grounding for all 13 primitives. Takes a **model, not a TaskContext**, so it
    runs on the dev box with only OpenRouter (no robot/CUDA import chain).
    Includes a no-robot dry run: `python -m tasks.GPSR.parse`.
  - `subtasks.py` — `ReceiveAndPlanCommands` parses each command and **speaks the
    plan**; `Command`/`CmdStatus` blackboard model. `ExecuteCommands` is Tier-2
    (agent fallback) for now. `run.py` wires the world model + `PeopleStore`.
  - **Tests:** `tests/test_gpsr_parse.py` — 25 offline unit tests (world,
    grounding per primitive, plan render) run with no LLM/robot.
    `tests/test_gpsr_coverage.py` — the **coverage gate**: feeds
    `tests/gpsr_command_corpus.txt` (every generator category) through the LLM
    parser, asserts ≥ `GPSR_COVERAGE_MIN` (0.85). **Measured 97% (33/34); the one
    miss — a superlative query ("the biggest object on the desk") — was then
    handled (generic-object placeholders ground as placement-scoped queries) and
    re-verified passing on the real LLM path via `python -m tasks.GPSR.parse`.**
    The split path (`parse_commands`) and the spoken plan were also verified on
    the real path with a three-commands-at-once utterance. The gate needs
    `OPENROUTER_API_KEY` (skips without it).
- **Phase 1 — NEXT:** `skills.py` + deterministic Tier-1 dispatch of each
  `PlanStep` to a skill (navigate/find/count/say/greet first — no arm), reusing
  HRI people skills; replace `ExecuteCommands`' agent-only path with
  dispatch-then-fallback.
- **Phase 2 / 3 —** manipulation (Restaurant reuse), then interleave bonus.

### Known gaps / decisions deferred to Phase 1+
- **Real poses** — `world.toml` poses are all `[0,0,0]`; navigation is a no-op
  until the announced arena is filled in.
- **Synonym robustness** — the corpus uses in-vocab nouns; out-of-vocab synonyms
  ("coke"→cola) currently ground as gaps. Measure on rephrased variants and
  decide whether to add a synonym layer or lean on the LLM to normalize.
- **`say`/`tell` knowledge source** (above) — unbuilt; `say` steps carry the info
  string but nothing answers open questions yet.
